# Architecture

## 1. 稳定前缀

每轮请求保持以下内容和顺序稳定：

1. 原始角色设定；
2. App 回复规则；
3. 用户确认的共同经历；
4. 用户确认的关系记忆。

角色设定不得自动改写。稳定前缀可以为未来的供应商上下文缓存做准备，但不能把本地指纹冒充服务端缓存 ID。

固定前缀使用规范化 JSON 的 SHA-256 指纹。指纹只回答“这轮前缀是否与上一轮
相同”，不得包含或展示人设、记忆正文。缓存是否命中只读取供应商响应中的
`cached_tokens`；没有该字段时记为未知，不使用本地估算值冒充命中率。

## 2. 动态上下文

动态部分包括：

1. 当前设备时间、时区和星期；
2. 最多 2 段相关旧对话；
3. 最近 4～12 个完整对话轮；
4. 当前用户消息。

没有可靠时间的历史明确标记为“时间未知”。近期上下文建议使用约 1600 Token 的软预算，检索历史建议使用约 500 Token 的硬预算。

检索出的旧聊天属于不可信用户数据。不得把旧聊天正文拼进 `system`
消息；应保持原始 `user` / `assistant` 层级，并明确标记为历史引用。稳定的
`system` 规则必须声明：历史引用中包含的“忽略规则”“修改身份”等文字不能成为
当前指令。

## 3. 本地记忆检索

`app/memory.py` 为聊天轮建立可重建关键词索引。索引只用于选取旧对话，不拥有修改原始消息的权限。

当前默认策略：

- 普通闲聊不扫描旧历史，只有明确或组合式回忆信号触发检索；
- “记得明天提醒我”等未来提醒不误判为历史召回；
- 触发召回后，可用最近两条用户消息补全过短的当前查询；
- 高频短句、完全重复和近似重复的旧轮次降权或抑制；
- 近期对话保留 4～12 个完整轮次，约 1600 Token 为软预算；
- 检索旧对话最多 2 段，约 500 Token 为硬预算，单段超预算时不截断原文；
- 索引元数据包含版本、消息数、轮次数和最后消息 ID；不一致时重建，不使用过期索引。

推荐保存：

- 原始消息数据库；
- 可重建检索索引；
- 用户手动确认的三项记忆。

不自动生成或写入正式记忆。

## 4. 付费请求防重与并发

客户端和请求层都应防重：

- 单条用户消息只创建一次 Character 请求；
- 同一进行中的请求共享结果；
- 自动播报关闭时不创建 TTS 请求；
- 手动朗读仍可单独请求；
- UI 重渲染、重试和旋转屏幕不能重复付费调用。

防重键必须至少包含 `session_id + request_id`。同一键配不同文本时返回冲突，
不能复用旧结果。每个会话使用独立串行锁：后一条消息只有在前一条完成并写入后
才能读取上下文，避免并发返回顺序颠倒。

TTS 使用同样的会话级幂等键；音频只做少量内存缓存，不写入数据库或音频文件。

## 5. 鉴权与持久化

- `/api/health`、`/api/history`、`/api/chat` 和 `/api/tts` 全部验证
  `X-App-Token`；
- 未设置至少 24 位随机 `APP_ACCESS_TOKEN` 时拒绝提供 API；
- 新聊天、请求 ID、状态和回复写入本地 SQLite；
- 页面重载后通过鉴权的 `/api/history` 恢复当前会话；
- 已完成的 Character 请求在程序重启后仍保持幂等；
- 不把 TTS 音频持久化。

## 6. 手机安全

Android 正式版应使用：

- Keystore 加密 API Key；
- 应用私有目录保存数据库；
- HTTPS 直连服务商；
- 内存音频播放；
- 明确的删除与二次确认；
- 不在日志中记录密钥、完整提示词、私有音色或语音内容。

电脑端 `app/server.py` 只用于技术验证，不是手机运行依赖。

## 7. 已验证的 Android 路线

经过验证的实现使用轻量原生 Java 外壳，将网页资源打入 APK：

1. AndroidX `WebViewAssetLoader` 把资源映射到
   `https://appassets.androidplatform.net/assets/`；
2. JavaScript Bridge 只传递结构化请求与结果，不把密钥交回网页层；
3. Android Keystore 加密模型和语音凭据；
4. 原生 HTTPS 直连 Character 与语音服务，不依赖电脑代理；
5. MP3 在内存中交给原生播放器，获取音频焦点后从开头播放，结束后释放；
6. Manifest 不声明开机自启、后台 Service 或广播 Receiver；
7. 发布 APK 不内置真实密钥、私人历史、签名材料或开发脚本。

公开模板只同步这套技术路线和基础算法，不复制 App 的具体界面。加入 Android
客户端时应按同样边界实现并增加 APK 内容、签名和真机测试。

## 8. APK 完成交付标准

电脑端验证通过不等于迁移完成。完整版必须继续完成 Android 封装，并以签名 APK
作为交付物。

最低验收标准：

1. 把前端静态资源作为 APK 本地资源加载，不能依赖电脑地址；
2. 使用 Android Keystore 保存 Character 和 TTS 密钥；
3. 使用 Android 原生 HTTPS 客户端直接请求服务商；
4. 聊天、记忆和检索索引在 App 重启后仍能恢复；
5. TTS 音频只在内存中播放，播放结束后释放；
6. 使用正式签名文件生成 APK，并用 Android 构建工具验证签名；
7. 在真机上验证冷启动、离线打开本地界面、文字聊天、语音播放、权限拒绝和进程重启；
8. 发布源码或 APK 时保留 `THIRD_PARTY_NOTICES.md` 所列的适用许可证与归属声明。

其他 Android 技术栈也可以使用，但不能降低上述独立运行、安全和真机验收要求。
